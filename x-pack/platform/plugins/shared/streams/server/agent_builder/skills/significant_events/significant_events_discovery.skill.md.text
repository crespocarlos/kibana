This skill provides canonical definitions and write tools for the Significant Events discovery workflow.

<sev_framework>

Criticality is a 0‚Äì100 integer measuring **user-experience impact**, not signal anomaly strength. `p_value` measures signal credibility; `criticality` measures blast radius and operational urgency.

| Range | Tier | Meaning |
|---|---|---|
| 76‚Äì100 | SEV1/2 | Core user journey broken or live data (PII/credentials) confirmed exposed. Page immediately. |
| 51‚Äì75 | SEV2/3 | Feature degraded, some user exposure, workarounds may exist. Respond within the hour. |
| 31‚Äì50 | SEV3/4 | Partial degradation, limited blast radius, workarounds exist. Schedule a fix. |
| 0‚Äì30 | SEV4/5 | Noise or confirmed false alarm. Demote or monitor. |

**p_value gate:** signal with `p_value > 0.1` ‚Üí cap criticality at 30, regardless of impact narrative.

**No-evidence cap:** if all evidence entries have `row_count = 0` or no queries were run ‚Üí cap at 50.

**When uncertain, choose the lower tier** (`acknowledged` over `promoted`, lower criticality over higher).

</sev_framework>

<confidence_scale>

Confidence is a float 0.0‚Äì1.0 measuring **root-cause correctness** ‚Äî how certain you are that the working theory is accurate. It does NOT measure signal strength (that is `p_value`) and does NOT measure user impact (that is `criticality`).

**Base from p_value (failure hypotheses):**
- `p_value ‚â§ 0.01` ‚Üí 0.55‚Äì0.65
- `p_value 0.01‚Äì0.05` ‚Üí 0.40‚Äì0.55
- `p_value 0.05‚Äì0.1` ‚Üí 0.25‚Äì0.40
- `p_value > 0.1` ‚Üí below 0.25

**Corroboration bonuses** (additive, max +0.10 each):
- ‚â•2 evidences with `row_count > 0` confirming the failure pattern
- KI backing present and causal chain aligns
- Failure spans ‚â•2 detection windows with no recovery sign

**Downward adjusters:**
- Queries with `row_count = 0` outnumber confirmed (‚àí0.10)
- No KI match found (‚àí0.10)
- Single evidence only (‚àí0.05)

**Hard caps:**
- All evidences have `row_count = 0` ‚Üí cap at 0.40
- No evidence calls made ‚Üí cap at 0.40
- No KI match for any cause ‚Üí cap at 0.65
- 0.90+ requires: `p_value ‚â§ 0.01`, ‚â•2 confirmed evidences, KI backing

**`cause_kis` ceiling:** without KI backing you cannot confirm which user journey is at risk. Cap at 0.65 when `cause_kis` is empty.

</confidence_scale>

<change_type_semantics>

| Value | Meaning | Likely causes |
|---|---|---|
| `spike` | Sudden increase in alert volume | Load surge, cascading failure, noisy rule |
| `dip` | Sudden decrease in alert volume | Service down (no data to alert on), rule disabled, data pipeline failure |
| `step_change` | Sustained level shift | Config change, new deployment, capacity change |
| `trend_change` | Gradual directional shift | Growing workload, degrading performance, slow leak |
| `distribution_change` | Overall distribution shifted | Mixed traffic pattern change, deployment rollout |
| `non_stationary` | No discrete change point, but not stationary | Gradual drift, chronic instability ‚Äî weak signal |
| `stationary` | No change point found ‚Äî distribution stable | Rule returned to normal, false positive, or noise |

A `dip` is especially important ‚Äî it often means a **service went DOWN** and stopped producing alerts. Investigate as service-down, not noise.

</change_type_semantics>

<ki_guidance>

Always call `search_knowledge_indicators` first ‚Äî before any `execute_esql`. One call for the entire batch, `kind: ["feature", "query"]`, with `stream_names` required.

KI types and their use:
- **Feature KI (`feature.type: "dependency"`)**: maps causal topology. `source ‚Üí target` shows service dependency. Build `dependency_edges` from these.
- **Feature KI (`feature.type: "infrastructure"`)**: shared infrastructure context. Build `infra_components` from these.
- **Query KI**: ES|QL queries pre-scoped to a detection rule (`rule.id` matches `rule_uuid`). These are the primary confirmation queries ‚Äî run exact-match KI queries for every `kind:'detection'` rule.

**Follow-up rules:**
- If 0 query KIs returned ‚Üí one follow-up call with `kind: ["query"]`, same `stream_names`.
- Deferred entity KI follow-up: after recording results, for domains with unconfirmed rules ‚Üí one `kind: ["feature"]` call.

**Derive `cause_kis`:**
1. Path 1 ‚Äî Query KI chain: for each exact-match KI whose query returned ‚â•1 row:
   a. **Structured (primary):** traverse its `features[]` array ‚Äî each entry has an `id` that references a feature KI slug. Look up any feature KIs with matching `id` and `type: "entity"`. Collect these as candidates.
   b. **Prose (fallback):** if `features[]` is absent or empty, scan its `evidence[]` array for `"Entity:"` entries and find the matching entity KI by name.
   c. **Topology filter (apply after collecting candidates):** cross-reference `dependency_edges` to remove victim/caller entities. For each candidate entity: if the service appears only as `source` (never as `target`) in `dependency_edges`, AND the confirmed failure pattern is a connectivity failure (connection refused, unreachable, timeout dialing the target), it is a caller experiencing the failure ‚Äî NOT a cause. Remove it. Keep only entities that appear as `target` in the relevant edge, or that have no dependency edge context (no false negatives). Prefer fewer, accurate cause KIs over broad attribution.
2. Path 2 ‚Äî Continuation inheritance: when Path 1 produced nothing and this is a continuation, inherit from the matched open episode.
3. Always emit `cause_kis: []` when no path produces candidates.

</ki_guidance>

<discovery_tools>

<tools>
- **search_knowledge_indicators** ‚Äî query the service knowledge base (feature + query KIs). Required before any `execute_esql` call.
- **execute_esql** ‚Äî run an ES|QL query to confirm or refute a KI hypothesis. KI-scoped queries only ‚Äî no freeform queries.
- **event_search** ‚Äî search significant events by status/stream. Set `include_episodes: true` to also return linked open discovery documents; use this for episode continuation checks before calling `discovery_write`.
- **discovery_write** ‚Äî append a discovery document (immutable data stream; latest-source pattern resolves to the most recent version per slug). Kinds: `discovery` | `clearance` | `handled`. For new episodes, omit `discovery_slug` ‚Äî the tool generates a unique slug from stream and rule names + a UUID8 suffix and returns it. For continuation and clearance, pass the candidate's `discovery_slug` verbatim. Deduplication is applied automatically with a default `now-1h` window; override `dedup_window` only when a different window is explicitly needed.
- **events_write** ‚Äî create or version a significant event for a discovery episode (workflow write path). Skips write if status is unchanged; chains new version via `previous_event_id`.
</tools>

**`discovery_write` schema:**

```json
{
  "kind": "discovery",
  "title": "Auth service ‚Äî login endpoint: connection refused",
  "summary": "...",
  "root_cause": "...",
  "impact": "...",
  "rule_names": ["High HTTP 5xx Error Rate"],
  "stream_names": ["logs.auth"],
  "criticality": 82,
  "confidence": 0.74,
  "detections": [...],
  "cause_kis": [...],
  "evidences": [...]
}
```

- `kind: "discovery"` ‚Äî new or continuing episode. Omit `discovery_slug` for new episodes; pass the candidate's slug verbatim for continuation.
- `kind: "clearance"` ‚Äî all rules quiet; pass the matched episode's `discovery_slug` verbatim.
- `kind: "handled"` ‚Äî stamp an episode as processed after calling `events_write`.

**`events_write` schema:**

```json
{
  "discovery_slug": "auth-service__high-http-5xx-error-rate-ab12cd34",
  "discovery_id": "...",
  "status": "promoted",
  "stream_names": ["logs.auth"],
  "title": "Auth service ‚Äî login endpoint: connection refused",
  "summary": "...",
  "root_cause": "...",
  "criticality": 82,
  "confidence": 0.74,
  "recommendations": ["..."],
  "assessment_note": "...",
  "cause_kis": [...],
  "dependency_edges": [...],
  "infra_components": [...],
  "evidences": [...]
}
```

- `cause_kis`, `dependency_edges`, `infra_components`: echo verbatim from the input discovery ‚Äî read-only, never modify.
- `evidences[]`: the merged evidence trail ‚Äî copy from the discovery's `evidences[]`, adding `confirmed: true` on entries you re-ran with `execute_esql` this cycle. Carry forward unverified entries unchanged. Never omit the array; never emit `[]`. Per entry: `rule_name`, `rule_uuid`, `stream_name`, `result`, `row_count`, `esql_query`, `description`, `collected_at`. Omit `confirmed` on unverified entries; never set `false`.

Returns `{ event_id, discovery_slug, status, written }`. Use `event_id` for downstream workflow steps (e.g. `trigger_investigation`).

`event_create` is the chat-path tool for operator-initiated events. `events_write` is the workflow-path tool for assessment and escalation decisions.

</discovery_tools>

<significant_event_narrative_standards>

**`summary`** ‚Äî a complete, self-contained incident brief readable cold. Four elements in order:
1. **Service + operator-visible symptom** ‚Äî what the user or dependent service experiences ("requests are failing", "users cannot authenticate"). Not "anomaly detected", "alert fired", "change point observed".
2. **Who is affected / blast radius** ‚Äî who cannot do what, and which exposed path carries the impact. Name the exposed path when `dependency_edges` has `exposure: "exposed"` entries.
3. **Magnitude and recovery status** ‚Äî error rate or count, onset time, whether it is recovering or stable.
4. **Context-specific close** ‚Äî provide what is most useful at your stage: when writing a discovery brief, end with a confidence signal ("Confidence {float} ‚Äî {one-phrase reason}"); when writing a significant event (assessment/escalation), end with the single most time-sensitive on-call action.

**Format**: `"{Service}: {symptom}. {Who/blast radius}. {Magnitude, onset, recovery}. {Confidence signal or most urgent action}."`

**Example (discovery brief)**: `"Auth service: login requests failing with connection refused. Users cannot authenticate; API gateway is exposed. ~200 errors/min, onset 14:22 UTC, no sign of recovery. Confidence 0.74 ‚Äî KI-confirmed, 3 found evidences."`

**Example (significant event)**: `"Auth service: login requests failing with connection refused. Users cannot authenticate; the API gateway is exposed and returning errors to clients. ~200 errors/min since 14:22 UTC, no recovery. Immediately check auth-service pod health and upstream session store connectivity."`

**Never**: detection mechanics as the lead (rule names, stream names, `alert_count`, `change_point_type`, `p_value`); causal mechanism detail; error codes as primary content; raw log content. Root cause belongs in `root_cause`, not here.

**`root_cause`** ‚Äî lowest actionable cause. State as a direct claim, not a symptom.
- **Sentence frame**: `"[Service] is [failing / timing out / erroring] because [mechanism] is [exhausted / unreachable / misconfigured / returning X]."`
- The "because" clause is what makes it actionable: it names the mechanism an SRE can act on. A theory without a "because" clause is a symptom, not a cause.
- Ground in evidence (cited KI + row count, or confirmed query results). State directly ‚Äî uncertainty belongs in `confidence`, not in prose hedges.
- **Never**: "queries show/confirm patterns", "consistent with the detection", "results indicate the presence of".

**`recommendations`** ‚Äî 3 candidate stop-the-bleeding actions ranked by likelihood of stopping user impact. The SRE will try them in order until impact stops.
- Order: most reversible and most targeted first (highest confidence based on evidence), broader or less certain interventions last.
- Each action: name the specific service, endpoint, or error code, and include the exact operational command (`kubectl`, `helm`, `systemctl`, CLI). Act-now only ‚Äî never include ES|QL or diagnostic queries.
- Prefer reversible over irreversible. Prefer scoped (one service, one flag) over broad (restart all, redeploy everything).
- Actions that address the `root_cause` directly rank above those that only reduce exposure.

</significant_event_narrative_standards>

<discovery_output_standards>

**Output discipline** ‚Äî applies to all narrative fields in every tool call:
Never reproduce payload values, PII, tokens, UUIDs, or raw log content. Use only: service names, error codes/types, field paths, counts, onset timing (hour/minute precision). Cite KIs by human-readable name, entities by service name.

**Shared array field formats** ‚Äî emit these consistently in every `discovery_write` call:

- `cause_kis` ‚Äî always emit; `[]` when no path produced candidates. Each entry: `name` (from `feature.id`), `stream_name` (from `feature.stream_name`). Never add a `confirmed` field.
- `dependency_edges` ‚Äî always emit; `[]` when empty. Each entry: `source`, `target`, `protocol` (omit if absent), `exposure`.
- `infra_components` ‚Äî always emit; `[]` when empty. Each entry: `title` (omit if absent), `workloads`, `exposure`.
- `evidences` ‚Äî always emit; `[]` only when no tool calls were made. Each entry: `rule_name`, `rule_uuid`, `stream_name`, `result`, `row_count`, `esql_query` (exact string as sent; `null` for quiet-rule dispositions), `description`, `collected_at`. No payload values in `description`.

</discovery_output_standards>
