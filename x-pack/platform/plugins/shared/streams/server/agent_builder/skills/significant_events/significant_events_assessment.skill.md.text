<scoring>

**`criticality`** measures **user-experience impact** (0–100 integer) — not signal anomaly strength. Consult the `sev_tiers` referenced file for tier ranges and override caps (call `read_file` with the path from `load_skill`'s `referenced_files` response).

**`confidence`** measures **root-cause correctness** (float 0.0–1.0) — not signal strength and not user impact. Consult the `confidence_scale` referenced file for p_value base ranges, corroboration bonuses, downward adjusters, and hard caps (call `read_file` with the path from `load_skill`'s `referenced_files` response).

</scoring>

<narrative_standards>

**`title`**: `"<Service> — <component>: <symptom>"` — stable and specific. `<component>` = affected subsystem (e.g., "write API", "login endpoint"); `<symptom>` = failure mode (e.g., "connection refused", "timeouts"). Must uniquely identify the incident — too generic if two different incidents could share it. No IPs, measurements, counts, or cycle numbers. For `kind:'clearance'`: recovery framing (e.g., `"Auth service — login endpoint: alerts recovered"`).

---

**`summary`** — a complete, self-contained incident brief readable cold. Four elements in order:
1. **Service + operator-visible symptom** — what the user or dependent service experiences ("requests are failing", "users cannot authenticate"). Not "anomaly detected", "alert fired", "change point observed".
2. **Who is affected / blast radius** — who cannot do what, and which exposed path carries the impact. Name the exposed path when `dependency_edges` has `exposure: "exposed"` entries.
3. **Magnitude and recovery status** — error rate or count, onset time, whether it is recovering or stable.
4. **Context-specific close** — provide what is most useful at your stage:
   - When writing a **discovery brief**: end with a confidence signal — `"Confidence {float} — {one-phrase reason}"`.
   - When writing a **significant event** (assessment/escalation): end with the single most time-sensitive on-call action.

**Format**: `"{Service}: {symptom}. {Who/blast radius}. {Magnitude, onset, recovery}. {Confidence signal or most urgent action}."`

**Example (discovery brief)**: `"Auth service: login requests failing with connection refused. Users cannot authenticate; API gateway is exposed. ~200 errors/min, onset 14:22 UTC, no sign of recovery. Confidence 0.74 — KI-confirmed, 3 found evidences."`

**Example (significant event)**: `"Auth service: login requests failing with connection refused. Users cannot authenticate; the API gateway is exposed and returning errors to clients. ~200 errors/min since 14:22 UTC, no recovery. Immediately check auth-service pod health and upstream session store connectivity."`

**Never**: detection mechanics as the lead (rule names, stream names, `alert_count`, `change_point_type`, `p_value`); causal mechanism detail; error codes as primary content; raw log content. Root cause belongs in `root_cause`, not here.

---

**`root_cause`** — lowest actionable cause. State as a direct claim, not a symptom.

- **Sentence frame**: `"[Service] is [failing / timing out / erroring] because [mechanism] is [exhausted / unreachable / misconfigured / returning X]."`
- The "because" clause names the mechanism an SRE can act on immediately. A theory without a "because" clause is a symptom, not a cause.
- Ground in evidence (cited KI + row count, or confirmed query results). State directly — uncertainty belongs in `confidence`, not in prose hedges.
- **Never**: "queries show/confirm patterns", "consistent with the detection", "results indicate the presence of".
- Omit when `confidence` < 0.5.

---

**`recommendations`** — 3 candidate stop-the-bleeding actions ranked by likelihood of stopping user impact. The SRE will try them in order until impact stops.

- Order: most reversible and most targeted first, broader or less certain interventions last.
- Each action: name the specific service, endpoint, or error code, and include the exact operational command (`kubectl`, `helm`, `systemctl`, CLI). Act-now only — never include ES|QL or diagnostic queries.
- Prefer reversible over irreversible. Prefer scoped (one service, one flag) over broad (restart all, redeploy everything).
- Actions that address the `root_cause` directly rank above those that only reduce exposure.

</narrative_standards>

<output_discipline>

Applies to all narrative fields in every tool call:

**Never reproduce**: payload values, PII, tokens, UUIDs, raw log content, or detection mechanics (`p_value`, `change_point_type`) — those are internal signal metadata, not operator-facing context. Use only: service names, error codes/types, field paths, counts, onset timing (hour/minute precision). Cite KIs by human-readable name, entities by service name.

</output_discipline>
